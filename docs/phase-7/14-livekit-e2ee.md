# 14 — LiveKit E2EE for Calls

Voice and video calls of ≤ 8 participants are end-to-end encrypted via LiveKit's Insertable Streams API. The SFU forwards ciphertext frames it cannot decode. Calls of > 8 fall back to `sfu_trust` mode where the SFU handles plaintext frames; UI is explicit about this.

## What Insertable Streams gives us

Modern WebRTC (and LiveKit's SDK) exposes a per-frame transform pipeline. We tap in:

- **Sender side:** receive plaintext frames from the encoder, encrypt them, forward to the SFU.
- **Receiver side:** receive ciphertext frames from the SFU, decrypt, forward to the decoder.

The SFU sees only ciphertext. It does not decode video, mix audio, or transcode — it forwards opaque packets.

## Per-call key

Every call has a single symmetric key. All participants share it. Frames are encrypted with this key.

```
call_key = csprng.fill_bytes(32)  // generated by the call initiator
```

The key is distributed via the standard E2E messaging pipeline:

```
Initiator:
  1. Generate call_key.
  2. For each invited participant device:
       wrapped_call_key = encrypt_via_double_ratchet(call_key, session_with_device)
  3. Send a "call invite" message to each invited participant via routing.SendEnvelope.
       The invite contains: call_id, livekit_room, mode, wrapped_call_key (specific to this device).
  4. Initiator joins LiveKit with their JWT.

Participant:
  1. Receive call invite via standard E2E.
  2. Unwrap call_key.
  3. Join LiveKit with the JWT issued by the call service.
  4. Configure Insertable Streams encrypt/decrypt with call_key.
```

## Per-frame encryption

For each outbound frame:

```
frame_key   = HKDF(call_key, salt=zero, info="velix.livekit.frame.v1", len=32)
frame_nonce = csprng.fill_bytes(24)
ciphertext  = AEAD-XChaCha20-Poly1305(
                  key=frame_key,
                  nonce=frame_nonce,
                  aad=frame_metadata,
                  plaintext=frame_payload)

wire_frame  = version_byte || frame_nonce || ciphertext
```

`frame_metadata` includes the codec, frame type (key/delta), and a per-stream identifier. Authenticating the metadata via AAD prevents frame substitution attacks.

The frame_key derivation collapses to AEAD with a fresh nonce per frame; the call_key drives the construction. We don't ratchet per-frame because:
- Frame rate is high (30/60 fps); ratcheting per frame is wasteful.
- The call_key rotates on participant change (the dominant attacker scenario).
- An attacker who steals call_key can decrypt frames until the next rotation regardless of per-frame ratcheting.

If a future audit requires per-frame ratcheting, we add it — the libsignal symmetric-ratchet primitives apply trivially.

## Key rotation on participant join

When a new participant joins:

1. The call's "owner" (initiator, or the longest-attending participant if owner left) generates a new call_key.
2. Distributes to all current participants + the new joiner via the E2E channel.
3. After all participants confirm receipt, owner switches to the new key for outbound frames.
4. There's a brief (≤ 1 s) window where some participants might still encrypt to the old key while receivers expect the new. We tolerate this via a 2-key "current and previous" buffer on receivers (libsignal's ratchet skip handling).

The new joiner cannot decrypt frames sent before their join (forward secrecy from their perspective).

## Key rotation on participant leave

When a participant leaves:

1. The call owner generates a new call_key.
2. Distributes to all remaining participants only.
3. Switches outbound encryption.

The leaving participant cannot decrypt frames after their departure (post-leave secrecy).

## Authentication of frames

Each frame's AEAD provides integrity (no undetected tampering). Beyond that, we do NOT sign each frame with a per-participant signature. Reasons:

- Per-frame signatures are expensive (Ed25519 per frame at 30 fps = 30 sigs/sec per stream).
- The call_key authenticates participation: only invited participants have it.
- The application-level "who's speaking" indicator is informational, not security-critical.

If two participants share a call_key and one wants to impersonate the other, they can — but they're already in the call together. We do not promise per-participant authenticity *within* a call.

If this becomes a concern (e.g., for high-stakes calls), participants verify each other out-of-band via safety numbers, the same way 1:1 conversations are verified.

## SFU trust mode (> 8 participants)

For 9+ participants, LiveKit's SFU needs to decode frames to do server-side mixing or simulcast. We do NOT use Insertable Streams in this mode.

Calls in `sfu_trust` mode are clearly indicated in the UI:

- Header banner: "12 people · server-aided"
- Per-frame trust state: same as LiveKit's default WebRTC (TLS to SFU; SFU sees plaintext).

The user is informed at call invite time. We do not silently downgrade.

## What LiveKit sees in `e2ee` mode

- The fact of a call (room name = call_id).
- Participant join/leave events.
- Frame timing (encoded frame rate).
- Frame size.
- Codec metadata.

Specifically not:

- Frame content (audio or video).
- Speaker identity (per-frame).
- Lip-sync data.

## What LiveKit sees in `sfu_trust` mode

Everything LiveKit sees in `e2ee` mode, plus:

- Decoded audio and video frames (transient, in-process).

LiveKit operators can choose to record this. We do NOT record. We do not configure LiveKit's recording features.

## Per-call ephemeral keys

The call_key is per-call. It is generated fresh for every call and discarded after the call ends. It does not derive from any long-lived call key.

`call_key` is held in RAM only for the duration of the call. On call end, all participants zeroize their copies.

## Key distribution via E2E

The wrapped call_keys are sent through the standard `routing.SendEnvelope` flow as a special message kind:

```
message body for call_invite: {
  kind: "call_invite",
  call_id,
  livekit_room,
  livekit_jwt,         // already issued for this participant
  mode,
  wrapped_call_key,    // encrypted to this recipient device's session
  expires_at
}
```

If a participant doesn't have a session with the call initiator, X3DH runs first. This is the same flow as a normal first-message-to-a-new-peer, just inside a call invite.

## Call recording

Velix does not record calls. Server-side. We do not have an architectural feature for it. Specifically:

- LiveKit's recording API is not enabled in our deployment.
- The call service does not retain any call data beyond metadata (start, end, participants).
- Per-frame logs are not stored.

If a user wants to record their own call, they use OS-level recording. We cannot prevent that, but we don't facilitate it.

## Failure modes

| Failure | Behavior |
|---|---|
| Participant fails to receive call_key | Joins LiveKit, but their decrypt fails; UI shows "Cannot decrypt this call". User leaves; rejoins after re-receiving the wrapped key. |
| Sender's encrypt fails | Frame dropped; metric incremented. Brief audio/video glitch. |
| Receiver's decrypt fails on a frame | Frame dropped. Brief glitch. |
| Key rotation fails to reach a participant | They drop frames until they reconnect and receive the new key. |
| Network loss during call | LiveKit handles reconnect; the call_key persists across reconnects. |
| Participant device compromised | The compromised participant has the call_key. Future calls require re-key. |

## Performance

- Per-frame encrypt/decrypt: ≤ 0.3 ms on iPhone 12 (XChaCha20 is ChaCha-fast).
- Total CPU at 720p30 + 1 audio: ~5% additional.
- Battery cost: marginal (~0.5% / hour additional vs unencrypted call).

## Banned

- Disabling Insertable Streams in `e2ee` mode.
- Server-side recording via LiveKit's record API.
- Sharing the call_key with the call service.
- Per-frame plaintext traversal of the SFU in `e2ee` mode.
- Reusing call_key across calls.
- Persisting call_key to disk.
- Signing all frames at 30+ fps with Ed25519 (cost-prohibitive; banned because it's a footgun for someone to "fix" performance issues).
