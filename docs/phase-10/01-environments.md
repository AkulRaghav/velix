# 01 — Environments

Three environments. Strict separation. Different cloud accounts, different secrets, different domains, different data.

## Dev

The engineer's local machine. Runs against `docker-compose` for everything that's not the Flutter app itself.

```yaml
# ops/dev/docker-compose.yml
version: "3.9"

services:
  postgres:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: velix_dev
      POSTGRES_PASSWORD: velix_dev_pw
      POSTGRES_DB: velix_dev
    ports: ["5432:5432"]
    volumes: ["pgdata:/var/lib/postgresql/data"]

  redis:
    image: redis:7-alpine
    ports: ["6379:6379"]

  nats:
    image: nats:2.10-alpine
    command: ["-js", "-m", "8222"]
    ports: ["4222:4222", "8222:8222"]

  minio:
    image: minio/minio:latest
    command: server /data --console-address ":9001"
    environment:
      MINIO_ROOT_USER: dev
      MINIO_ROOT_PASSWORD: dev-secret
    ports: ["9000:9000", "9001:9001"]
    volumes: ["miniodata:/data"]

  livekit:
    image: livekit/livekit-server:latest
    command: ["--dev"]
    ports: ["7880:7880"]

volumes:
  pgdata:
  miniodata:
```

Engineers run `make dev-up` to start; `make dev-down` to stop. Database is reset frequently; migrations re-run.

Engineers do not have access to staging or production from their local environment. Even read-only is forbidden — they go through the proper paths (CI logs, Grafana, audited shells).

## Staging

A **single-cell, production-shaped** environment. Different AWS account from production. Cell topology = `us-east-1` only.

| Property | Staging | Production |
|---|---|---|
| Cloud account | `velix-staging-12345` | `velix-prod-67890` |
| Domain | `staging.velix.app` | `velix.app` |
| Postgres | RDS Aurora staging cluster | RDS Aurora primary + 2 replicas per cell |
| Redis | ElastiCache 3-node | ElastiCache 6-node cluster per cell |
| NATS | 3-node cluster | 5-node cluster per cell |
| LiveKit | 1 node | 3-10 autoscale per cell |
| Cells | 1 | 3+ |
| Data | Synthetic; no production data | Real users |
| TLS | Let's Encrypt staging | Let's Encrypt production + cert-manager |
| Secrets | Vault staging instance | Vault production instance |
| iOS / Android builds | TestFlight / Play Internal Testing | App Store / Play Store |

Staging mirrors production architecturally. Staging traffic is generated by:

- CI integration tests on every main merge.
- Synthetic probes (every 60 s).
- Engineer manual validation of new features.
- A small group of internal beta testers (Velix employees + close design partners).

Staging is cleared and re-seeded weekly. We do not preserve test data across weeks.

## Production

The real environment. Multi-cell:
- `prod-us-east-1`
- `prod-eu-west-1`
- `prod-ap-southeast-1`

Each cell is a separate Argo CD `Application` deploying the same Helm charts with cell-specific values.

Production access is **dual-control**: any human action (kubectl exec into a pod, manual database query, etc.) requires two engineers to approve via a tooling pipeline. We do not give individual engineers production credentials.

The exceptions:
- On-call engineers have time-limited break-glass access (≤ 4 hours, audit-logged, page-triggered).
- Read-only access to logs and metrics via Grafana is normal.

## Configuration per environment

We use environment-specific values via Helm:

```yaml
# ops/helm/velix-routing/values-prod-us-east-1.yaml
replicaCount: 3
postgres:
  host: prod-routing-us-east-1.cluster-xyz.rds.amazonaws.com
  database: velix_routing
  user: routing_service     # Vault-issued, short-lived
nats:
  cluster: nats.prod-us-east-1.svc.cluster.local
ratelimit:
  send_envelope_per_account_per_minute: 60
livekit:
  jwt_audience: livekit-us-east-1.velix.app
otel:
  endpoint: otel-collector.observability.svc.cluster.local:4317
```

```yaml
# values-staging.yaml
replicaCount: 1
postgres:
  host: staging-routing.cluster-abc.rds.amazonaws.com
  database: velix_routing
  user: routing_service
nats:
  cluster: nats.staging.svc.cluster.local
ratelimit:
  send_envelope_per_account_per_minute: 600   # generous for testing
otel:
  endpoint: otel-collector-staging.observability.svc.cluster.local:4317
```

The Helm chart structure is identical; only values differ. This makes promotion mechanical.

## Promotion paths

```
[engineer's branch]
        │
        ▼
[main]                          ← merged after PR approval
        │
        ▼
[staging]                       ← auto-deployed via Argo CD on every main merge
        │
        ▼
[prod-us-east-1 canary 5%]      ← manual approval
        │
        ▼
[prod-us-east-1 100%]           ← if canary green for 30 minutes
        │
        ▼
[prod-eu-west-1 canary 5%]
        │
        ▼
[prod-eu-west-1 100%]
        │
        ▼
[prod-ap-southeast-1 canary 5%]
        │
        ▼
[prod-ap-southeast-1 100%]
```

Each promotion gate has automated verifications + a human approver. The approver in production is the on-call engineer (and a second engineer for production releases).

## Configuration drift

Tools that detect drift:

- **Argo CD's `OutOfSync` status**: if cluster state differs from git, Argo flags it.
- **Terraform state drift**: nightly `terraform plan` runs in staging and production; non-zero diffs alert.
- **Helm values diff**: a CI job verifies that `helm diff` between consecutive deploys is the expected diff.

Hand-edits to a cluster (kubectl edit, hand-applied yaml) are forbidden. Any change goes through git → CI → Argo CD.

## Cross-environment isolation enforcement

| Boundary | Enforcement |
|---|---|
| AWS accounts | Separate accounts; cross-account IAM forbidden |
| Postgres data | Different RDS clusters; firewall rules deny cross-env |
| Vault | Different Vault instances; different unseal keys |
| Domain names | Different DNS zones |
| Image tags | Same images; different deployment tags |
| Telemetry | Separate Grafana orgs; staging data never reaches production dashboards |

Engineer access:
- Staging: any engineer (audit-logged).
- Production: dual-control through tooling; on-call break-glass.

## "What if I just want to test something quickly in production?"

Don't. Use:
- Feature flags (a kill-switch we can toggle without re-deploy).
- Canary cohorts (5% of users see the change; metrics watched).
- Staging — most "I just want to test" needs staging, not production.

If you genuinely need production access for incident diagnosis: page yourself, document the access, complete the action, write up the postmortem.

## Banned

- Engineer machines connecting to production resources.
- Production credentials in environment variables on local machines.
- Manual `kubectl apply` against any cluster.
- Manual Helm upgrades (Argo CD reconciles from git).
- Sharing production secrets via chat.
- Skipping staging for "small" changes.
- Promoting to production from a non-staging branch.
- "Read-only access" to production data for engineer convenience.
